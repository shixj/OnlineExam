## Context

当前仓库已经有完整的后端 API、后台管理端和 React H5 练习端。用户已经明确要求保留现有 H5，同时从零新增一套原生微信小程序客户端。第一阶段目标不是直接完成微信正式发布，而是先建立一个能够在微信开发者工具和真机中运行的原生小程序工程，并复用已有后端接口完成基础练习闭环。

约束：
- 当前后端接口以账号密码登录、JWT 鉴权、题库列表、题库详情、练习会话、错题、历史记录为核心。
- 不在第一阶段引入微信登录、手机号一键登录、订阅消息等开放平台能力。
- 原生小程序工程需要独立于现有 H5，不应破坏现有 `apps/app` 开发流程。
- 小程序端需要为手机优先，同时保留 iPad 可用性。

## Goals / Non-Goals

**Goals:**
- 建立独立的 `apps/miniapp` 原生微信小程序工程骨架。
- 提供统一的请求封装、登录态存储、全局配置与页面导航。
- 基于现有后端 API 打通登录、题库列表、题库详情、顺序练习、错题、历史、结果页的首版页面骨架。
- 为后续真机调试、发布配置和微信能力扩展留出清晰结构。

**Non-Goals:**
- 不在这一阶段完成微信开放能力接入，如 `wx.login` 换取 OpenID、手机号授权登录等。
- 不重构现有后端接口为小程序专用协议。
- 不替换现有 H5 练习端，也不要求本阶段功能完全追平 H5。
- 不在这一阶段处理正式发布所需的域名、备案、HTTPS、服务器域名配置。

## Decisions

### Decision: 新增独立原生小程序工程而不是改造现有 H5
- Rationale: H5 练习端与原生小程序运行时、页面结构、组件体系完全不同，直接改造会破坏现有浏览器端能力。独立工程可以最小风险并行演进。
- Alternatives considered:
  - 直接把 React H5 包进 `web-view`：不适合作为长期正式方案，且依赖企业小程序与业务域名。
  - 全量迁移并删除 H5：会丢失现有浏览器调试与跨设备访问价值。

### Decision: 第一阶段继续复用现有账号密码登录 API
- Rationale: 现有后端已具备完整认证流程，先复用可以最快形成可验证闭环。
- Alternatives considered:
  - 立即切换微信登录：实现成本更高，且需要额外的微信开放平台配置与服务端改造。

### Decision: 小程序工程采用原生目录结构与轻量公用工具
- Rationale: 目标是尽快形成可在微信开发者工具中运行的基础工程，避免引入额外框架增加迁移成本。
- Alternatives considered:
  - 使用跨端框架重新搭建：会让当前需求从“原生小程序”偏离到另一套技术栈迁移。

### Decision: 首版页面聚焦业务骨架与接口打通
- Rationale: 用户已经有可用的 H5 版本，原生端第一步最重要的是打通端能力和核心流程，而不是一次性追平全部交互细节。
- Alternatives considered:
  - 一次性覆盖所有高级交互：范围过大，延缓首个原生版本落地。

## Risks / Trade-offs

- [原生小程序与 H5 页面双端并行会增加维护成本] → 通过共享接口协议、共享文案与统一目录职责降低分叉风险。
- [账号密码登录不够“小程序原生”] → 将其视为第一阶段过渡方案，在后续变更中再引入微信身份能力。
- [当前后端返回结构以 H5 需求为主] → 先通过小程序端适配层消化，只有必要时才新增后端小程序专用字段。
- [真机与开发者工具表现可能存在差异] → 保留独立调试说明，并把手机/iPad 真机验收列入后续任务。

## Migration Plan

1. 新增 `apps/miniapp` 工程骨架与基础说明文件。
2. 搭建全局 `app`、请求层、登录态、页面注册与底部导航骨架。
3. 接入现有后端登录、题库列表与详情接口。
4. 接入练习、错题、历史、结果页最小闭环。
5. 在微信开发者工具和真机中验证基础流程可用。

Rollback strategy:
- 本变更完全新增，不替换现有 H5；若原生小程序端未准备好，可以仅停止使用 `apps/miniapp`，不影响现有系统。

## Open Questions

- 第二阶段是否直接升级到微信身份登录，而不是继续保留账号密码登录。
- 是否需要为小程序端新增更适合触控操作的分页答题卡与手势切换。
- 是否在原生小程序端引入统一 UI 组件库，还是保持原生组件与轻样式方式。
